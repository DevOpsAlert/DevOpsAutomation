# DevOpsAutomation
Автоматизация задач администрирования с использованием скриптов и инструментов DevOps.

Дипломный проект по специальности «Сетевое и системное администрирование» (ПГУС, 2026). Практическая реализация инфраструктуры как кода на базе Yandex Cloud: от ручных операций администрирования — к воспроизводимой, автоматизированной и мониторингуемой системе.

## Архитектура

Инфраструктура развёрнута в Yandex Cloud и состоит из четырёх узлов:

```
┌─────────────────┐      ┌──────────────────┐      ┌──────────────────┐
│  Admin-машина   │      │  Control node    │      │  Target-сервер   │
│  (Windows 10)   │ ───► │  (Ubuntu 22.04,  │ ───► │  (Ubuntu 22.04)  │
│                 │      │   Ansible)       │      │                  │
└─────────────────┘      └──────────────────┘      └──────────────────┘
                                  ▲
                                  │
                         ┌──────────────────┐
                         │  GitHub Actions  │
                         │  (CI/CD)         │
                         └──────────────────┘
```

- **Admin-машина** — Windows 10, администрирование через PowerShell-скрипты
- **Control node** — Ubuntu 22.04, управляет развёртыванием через Ansible
- **Target-сервер** — Ubuntu 22.04, целевая машина с задеплоенными сервисами (Nginx в Docker, UFW, fail2ban)
- **CI/CD** — GitHub Actions прогоняет линтеры при каждом push и триггерит развёртывание

## Стек технологий

`Bash` `PowerShell` `Ansible` `Docker` `GitHub Actions` `Nginx` `UFW` `fail2ban` `Yandex Cloud` `Telegram Bot API`

## Состав репозитория

| Файл | Назначение |
|---|---|
| `check_resources.sh` | Мониторинг ресурсов сервера (CPU, RAM, диск) |
| `backup_home.sh` | Резервное копирование домашних директорий |
| `update_system.sh` | Автоматическое обновление пакетов системы |
| `ssh_hardening.sh` | Усиление безопасности SSH-доступа |
| `create_user.ps1` | Создание и настройка пользователей (Windows) |
| `service_check.ps1` | Проверка состояния ключевых служб Windows |
| `playbook.yml` | Ansible-playbook: развёртывание Nginx в Docker, настройка UFW и fail2ban |
| `inventory.ini` | Инвентарь хостов для Ansible |
| `.github/` | Workflow'ы GitHub Actions — линтеры и CI/CD-пайплайн |

Все скрипты продублированы уведомлением в Telegram через бота **DevOpsAlert** — по завершении задачи в чат приходит статус выполнения.

## CI/CD-пайплайн

При каждом push в репозиторий GitHub Actions автоматически:
1. Прогоняет `shellcheck` по всем `.sh`-скриптам
2. Прогоняет `PSScriptAnalyzer` по `.ps1`-скриптам
3. Проверяет playbook через `ansible-lint`
4. При успешной проверке — деплоит изменения на target-сервер через Ansible

Пайплайн прошёл через пять итераций до стабильной работы, включая замену GitLab CI на GitHub Actions из-за ограничений региональной верификации.

## Результаты

- Сокращение времени рутинных операций администрирования: **~145 → ~6 минут**
- Подтверждённая идемпотентность автоматизации: три последовательных прогона показали **10 → 5 → 2** изменённых задачи
- **8 640** автоматических проверок состояния ресурсов в месяц
- Защита диплома — оценка **«отлично»**

## Как запустить

```bash
# Клонировать репозиторий
git clone https://github.com/DevOpsAlert/DevOpsAutomation.git
cd DevOpsAutomation

# Развернуть инфраструктуру через Ansible
ansible-playbook -i inventory.ini playbook.yml

# Запустить проверку ресурсов вручную
./check_resources.sh
```

Для работы требуется настроенный доступ по SSH к целевому серверу и Ansible ≥ 2.14 на control node.
